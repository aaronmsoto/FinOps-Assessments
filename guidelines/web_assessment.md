# Web-based Assessment UI

## Overview

Provide an additional assessment interface — a self-contained HTML+JS application — generated from the same YAML specifications and process that create the existing Excel assessment and other artifacts. This browser-based option offers benefits such as interactive questionnaire input, visual results dashboards, built-in branding, and universal accessibility without requiring specific desktop software.

### Motivation

The Excel-based assessment serves as a solid foundation and will continue to be supported. A browser-based alternative can complement it by offering a more interactive assessment experience, richer visualization of results, easier customization for any branding purposes, and broader accessibility for teams that may prefer a browser-based workflow.

### Approach

A single, fully self-contained HTML file (all CSS, JS, and assets inlined) that:

- Is generated by the `finopspp` CLI from the same YAML specs that produce the XLSX today
- Works offline in any modern browser with zero external dependencies
- Provides a two-tab experience — **Results** (poster view, default) and **Assess** (questionnaire) — with a gear icon for configuration
- Auto-saves progress to `localStorage` with JSON export/import for portability
- Produces print-friendly output for PDF delivery via `Ctrl+P`

## Architecture

### Separation of Concerns

The key architectural principle is that the CLI generates **only the data** — the UI template is a standalone HTML/CSS/JS application maintained independently in the repo. This separation means:

- **UI changes** (layout, styling, interactions, new features) can be made by editing the template files directly — no Python, no CLI rebuild, no spec knowledge required
- **Data changes** (new profiles, updated capabilities, scoring rules) flow through the normal `finopspp generate` pipeline
- **Consultants** receiving a generated assessment can even hand-edit the embedded JSON data block if they need to adjust titles, descriptions, or scoring without re-running the CLI

### Template Structure

The template files live in `tools/templates/html/` and are maintained as normal, readable source files — not generated code. Each file has a clear responsibility:

```
tools/templates/html/
├─ assessment.html          HTML shell: document structure, tab containers,
│                           placeholder for data injection
├─ css/
│   ├─ poster.css           Poster view: domain cards, capability bars, colors
│   ├─ assess.css           Questionnaire: form controls, progress indicators
│   ├─ config.css           Config panel: gear icon, slide-out, form layout
│   └─ print.css            @media print: page breaks, hidden controls
├─ js/
│   ├─ app.js               App shell: tab switching, initialization, event bus
│   ├─ scoring.js           Scoring engine: action→capability→domain→overall
│   ├─ poster.js            Poster view: render domains/capabilities, color scale
│   ├─ assess.js            Questionnaire: render actions, handle input by score type
│   ├─ config.js            Config panel: title/subtitle, visibility, weights
│   ├─ storage.js           Persistence: localStorage auto-save, JSON export/import, save HTML copy
│   └─ charts.js            Charts: radar, bar, distribution (Canvas API / SVG)
├─ readonly.html            Read-only template for Save As HTML (bundled into main file)
└─ sample-data.json         Sample dataset for development and testing
```

During development and maintenance, contributors work with these files as normal source code — with syntax highlighting, linting, version control diffs, and the ability to test by opening `assessment.html` in a browser with the sample data.

### Assembly Process

The HTML composer's job is simple assembly, not code generation:

1. Read the HTML shell template (`assessment.html`)
2. Read and inline all CSS files into a `<style>` block
3. Read and inline all JS files into `<script>` blocks
4. Inject the generated JSON data into a `<script id="spec-data">` block
5. Write the assembled single-file output

This is deliberately minimal — the composer is a "bundler," not a "generator." The template files are the source of truth for all UI behavior.

### Data Model (embedded JSON)

The CLI generates a JSON data block that contains everything the UI needs to render and score. This is the **only part** that requires the CLI — the rest is static template code.

```json
{
  "generatedAt": "2026-03-28T12:00:00Z",
  "generator": "finopspp 1.0.0",
  "profile": {
    "id": 1,
    "title": "FinOps Foundation",
    "version": "1.0.0"
  },
  "domains": [
    {
      "id": 1,
      "title": "Manage the FinOps Practice",
      "capabilities": [
        {
          "id": 6,
          "title": "Governance, Policy & Risk",
          "actions": [
            {
              "id": 102,
              "title": "Establish Cloud Governance Framework",
              "description": "...",
              "weight": 1.0,
              "scoreType": "binary",
              "scoring": [
                { "score": 0, "condition": "No governance framework" },
                { "score": 10, "condition": "Framework established" }
              ],
              "formula": null,
              "references": [],
              "supplementalGuidance": []
            }
          ]
        }
      ]
    }
  ]
}
```

The data block is human-readable JSON in a clearly marked `<script>` tag:

```html
<script id="spec-data" type="application/json">
{ ... }
</script>
```

This means a consultant who receives the generated file can:

- Open it in a text editor and find the data block
- Adjust titles, descriptions, or weights directly
- Add or remove actions manually
- Re-open in the browser with changes applied — no tooling required

### Data Validation

On load, the app validates the embedded JSON data block and displays a clear, friendly error banner if the data is malformed or structurally invalid. This protects against hand-edit mistakes by consultants and catches corrupted files early.

Validation checks:

- Valid JSON (parse without error)
- Required top-level keys present (`profile`, `domains`)
- Each domain has `id`, `title`, `capabilities` array
- Each capability has `id`, `title`, `actions` array
- Each action has `id`, `title`, `scoreType`, `scoring` array
- `scoreType` is one of the recognized values (`binary`, `bucket`, `linear`, `value`)
- Scoring entries have `score` (number) and `condition` (string)

On failure, the app renders a banner at the top of the page with a human-readable message indicating what's wrong and which element is affected. The rest of the UI does not render — a broken data block should not produce a silently broken assessment.

### Development Workflow

Working on the template files does **not** require `finopspp` or Python:

1. Edit any file in `tools/templates/html/`
2. Open `tools/templates/html/assessment.html` in a browser
3. The template loads `sample-data.json` (a checked-in sample dataset for development and testing)
4. Iterate on CSS/JS with browser dev tools

To test with real spec data:

```
finopspp generate assessment --profile="FinOps Foundation" --format=html
open "assessments/FinOps Foundation/FinOps Foundation Assessment.html"
```

The `sample-data.json` file contains a representative subset of a real profile — enough to exercise all score types, multiple domains, and edge cases. It serves as:

- Local development without the CLI
- Automated testing (if added later)
- Demonstrating the UI to stakeholders

## User Experience

The UI has two primary tabs — **Results** (the default view) and **Assess** — plus a gear icon (⚙) that opens a configuration panel. On initial load, the Results poster should render with sensible defaults and look polished before any assessment input has been provided.

### Configuration (⚙ Gear Icon)

A slide-out or modal panel accessible via a subtle gear icon in the header. Not a tab — it overlays the current view and closes when done. Settings persist to `localStorage` and export with JSON.

#### General

| Field | Default | Purpose |
|---|---|---|
| Title | "FinOps Maturity Assessment" | Displayed in header as the main title |
| Subtitle | "Measure progress over time against accepted standards" | Displayed in header below the title (optional) |

#### Capability Visibility

A checklist of all capabilities in the profile, grouped by domain. Unchecking a capability hides it from the Assess tab and grays it out in the Results poster. Capabilities excluded from assessment are excluded from scoring. This allows users to scope an assessment to a subset of the framework.

#### Action Weight Overrides

A table of all visible actions with editable weight values:

| Weight | Behavior |
|---|---|
| `0.0` | Action is hidden and excluded from scoring |
| `1.0` | Default — full weight in capability score |
| `0.0 < w < 1.0` | Partial weight (de-emphasized but still scored) |
| `> 1.0` | Boosted weight (emphasized in scoring) |

Changes to weights dynamically recalculate results. The original spec weights are shown for reference and weights initially come from the specification yaml files.

#### Data Management

- **Export JSON**: Download current state (config + all responses + domain priorities) as a `.json` file
- **Import JSON**: Load a previously exported state, replacing current data
- **Reset**: Clear all responses and config (with confirmation)

#### Future: Branding

A dedicated branding section is planned for a future phase. It will include:

- Consultancy name, logo URL (or base64 data URI), and brand color(s)
- Client name and logo URL
- These values will appear in the header, poster, and print output

This is deferred to keep the initial implementation focused on core assessment functionality.

#### Domain Priority Ranking

Each domain header displays an editable priority number. Users assign a 1–4 stack ranking to indicate organizational priority across the four domains. This is an input layer unique to the web assessment — it does not affect scoring but provides strategic context:

- Displayed as a small editable badge or dropdown on each domain card in the poster
- Stored alongside assessment data in `localStorage` and included in JSON export/import
- Visible in print output next to each domain title
- Duplicate ranks are allowed (ties) — the UI does not enforce uniqueness

#### Score Visualization

- **Color scale**: >= 7.5 use RGB(0, 198, 147); >= 5.0 use RGB(0, 150, 100); >= 2.5 use RGB(0, 100, 50); else use RGB(100, 100, 100) (configurable thresholds possible in future)
- **Capability cards**: Show capability name and score bar/fill
- **Domain summary**: Weighted average of capability scores within the domain
- **Overall score**: Weighted average across all domains, displayed prominently
- **Unscored state**: Before any assessment input, capabilities display in a neutral/grey state — the poster layout is fully visible but scores are blank

#### Charts

All charts rendered with inlined JS (Canvas API or inline SVG — no external charting library to keep the file self-contained):

- **Radar/spider chart**: One axis per domain, showing score vs. max
- **Bar chart**: Domain scores within the overall framework (click-to-drill)
- **Bar chart**: Capability scores within a selected domain (click-to-drill)
- **Score distribution**: Histogram of action scores across the assessment

#### Print View

`@media print` styles that produce a clean, branded PDF:

- Page 1: Title, subtitle, and poster view (the visual framework map with scores and domain priorities)
- Page 2+: Detailed scores by domain and capability
- Exclude navigation tabs, buttons, and interactive controls from print

### Save As Read-Only HTML

A "Save As HTML" button (available on the Results tab and/or in ⚙ Data Management) freezes the completed assessment into a new, self-contained HTML file that can be shared, archived, or presented — but not edited.

#### Purpose

After configuration and assessment are complete, the user saves the final state as a read-only HTML file. This is the deliverable: a polished, non-editable record of the assessment that a consultant can send to a client, attach to a report, or archive for future comparison.

#### What's Included

The saved file contains two tabs:

1. **Results** — The poster view with final scores, domain priorities, title/subtitle, charts, and attribution. Identical to the live Results tab but with no interactive controls.

2. **Details** — A read-only record of every action response and score, organized by domain and capability. This replaces the interactive Assess tab with a static table:

```
┌─────────────────────────────────────────────────────┐
│ Domain: Manage the FinOps Practice                   │
│                                                     │
│  Capability: Governance, Policy & Risk    Score: 7.5│
│  ┌─────────────────────────────────────────────────┐│
│  │ Action                        Response   Score  ││
│  │ Establish Cloud Governance    Yes          10   ││
│  │ Define Policy Exceptions      1 of 3        4   ││
│  │ Monitor Policy Compliance     No             0   ││
│  │ ...                                             ││
│  └─────────────────────────────────────────────────┘│
│                                                     │
│  Capability: FinOps Practice Operations   Score: 6.0│
│  ...                                                │
└─────────────────────────────────────────────────────┘
```

The file also includes:
- All CSS (including `@media print` styles)
- FinOps Framework attribution
- A "Generated on [date]" timestamp in the footer

#### What's Removed

- Configuration panel (⚙ gear icon)
- All form inputs (radio buttons, checkboxes, number fields, dropdowns)
- LocalStorage logic (no auto-save, no import/export)
- The editable Assess questionnaire (replaced by the read-only Details tab)

#### Technical Approach

Uses a separate read-only template rather than DOM cloning — cleaner, less fragile, and independently testable.

A lightweight read-only template (`readonly.html`) is bundled into the main assessment file (e.g., as a `<script type="text/html" id="readonly-template">` block or as a JS string constant). This template only knows how to render results + details from a JSON state blob. It has no interactive controls, no localStorage, and no configuration logic.

The save process:

1. Serialize the current state as a JSON blob: config (title, subtitle), all action responses and scores, computed capability/domain/overall scores, domain priorities, and hidden capabilities
2. Read the bundled read-only template
3. Inject the state JSON into the template's `<script id="assessment-state">` block
4. Create a `Blob` from the assembled HTML
5. Trigger a download via a dynamically created `<a download="filename.html">` element
6. Default filename: `{Title} - {YYYY-MM-DD}.html` (browser save dialog allows renaming)

The read-only template lives in the repo as `tools/templates/html/readonly.html` and is assembled into the main file during generation — the same bundler pattern used for the main template.

#### Constraints

- No server required — the `<a download>` pattern works from `file://` in all modern browsers
- The saved file must be fully self-contained (same single-file constraint as the source assessment)
- File size should be smaller than the source file (no interactive JS, no bundled readonly template)
- The read-only template can be developed and tested independently with a sample state JSON

## Scoring Engine

The JS scoring engine must replicate the Python scoring logic from `finopspp`:

1. **Action score**: Determined by user response mapped to the action's scoring table
2. **Capability score**: Weighted average of action scores within the capability (respecting weight overrides from Config)
3. **Domain score**: Weighted average of capability scores (equal weight per capability, or future configurable)
4. **Overall score**: Weighted average of domain scores

Hidden capabilities (unchecked in Config) and zero-weight actions are excluded from all aggregations. Unanswered actions are excluded (not zero — truly excluded from the denominator).

### Formula Parsing

For `bucket` score types, the formula text contains checklist items (lines starting with `*`) and a formula expression (e.g., `10*Ceil(x/3)`). The JS engine must:

1. Parse checklist items from the formula text
2. Count checked items (`x`)
3. Evaluate the formula expression to produce the score

## Technical Constraints

- **Single file**: All HTML, CSS, JS, and spec data in one `.html` file
- **No external dependencies**: No CDN references, no fetch calls, no server
- **Offline-first**: Must work from `file://` protocol (no service workers)
- **Browser support**: Modern evergreen browsers (Chrome, Firefox, Edge, Safari — last 2 versions)
- **File size target**: Aim for under 500KB for a full profile assessment
- **Accessibility**: Semantic HTML, keyboard navigation, ARIA labels on interactive elements
- **Responsive**: Usable on tablet-sized screens and above (1024px+)

### Composer role

The HTML composer is deliberately simple. It does not generate HTML or JS — it **assembles** the pre-existing template files with the generated data:

1. Receive resolved profile data from the existing generation pipeline
2. Serialize it as JSON
3. Read template files from `tools/templates/html/`
4. Inline CSS and JS into the HTML shell
5. Inject the JSON data block
6. Write the single-file output

The existing generation pipeline already resolves profiles into a full data model (domains, capabilities, actions with scoring). The HTML composer hooks into the same pipeline as the Excel composer, receiving the same resolved data.

## Success Criteria

1. Generated HTML file opens in a browser and renders all domains, capabilities, and actions
2. All score types (binary, bucket, linear, value) are interactive and score correctly
3. Scores match the Excel assessment for the same inputs
4. Config changes (hide capability, adjust weight) correctly update scoring
5. Assessment state survives browser refresh (localStorage)
6. JSON export can be imported on a different machine and resume
7. Print produces a clean, branded PDF
8. File is fully self-contained — works from `file://` with no internet
9. Data validation catches malformed JSON and shows a clear error on load
10. Save As HTML produces a functional read-only file with Results and Details tabs

## Scoring Parity Test

The JS scoring engine must produce identical results to the Python `finopspp` scoring engine for the same inputs. To verify this:

1. Maintain a test fixture: a set of action responses (covering all score types) with expected scores at action, capability, domain, and overall levels
2. The Python engine computes expected outputs from the fixture and writes them to a known file (or the test asserts against hardcoded values)
3. The JS engine is tested against the same fixture — either via a Node.js test runner or by loading the sample data in a browser test harness
4. Both engines must agree on all computed scores to within floating-point tolerance (< 0.01)

This can run as a manual check during development or be automated as a CI step if a test runner is added later.
